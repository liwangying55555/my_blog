# batch-run-local

给本地项目们安排一个“启动遥控器” 🎮

每天干活，就是在各个项目里面寻宝，先打开对应的项目文件，打开终端，跑起来。找到下一个项目，打开终端，跑起来.....

这个项目在哪来着？🤔  
这个项目启动命令是 `npm run dev`，还是 `npm run serve`？  
那个项目好像还要先切到 Git Bash？  

然后熟练地打开终端、`cd`、翻 `package.json`、复制命令、回车。

一次两次还好，天天这么来，就很像上班打卡前先做一套热身操：不难，但烦。😵‍💫

所以就有了这个小工具：[`batch-run-local`](https://www.npmjs.com/package/batch-run-local)。

一句话概括它：

> 把本地多个项目收进一个小本本里，需要启动哪个项目，就从列表里选一下，然后它帮你打开 Git Bash 执行对应的 `npm scripts`。🚀

## 1. 它适合解决什么问题？

如果你也有下面这些情况，那它应该会挺顺手：

- 本地项目很多，路径记不住 🗂️
- 每个项目的启动命令不一样，经常要翻 `package.json` 📦
- Windows 下习惯用 Git Bash 跑项目 🪟
- 想要一个简单的本地 Web 页面，点按钮就启动项目 🖱️
- 不想把配置写进项目里，只想自己电脑本地管理 🤫

它不是那种很重的项目管理平台，也不是部署工具。它更像一个放在桌面旁边的小遥控器：

“今天要跑博客项目？”  
点一下。啪，Git Bash 开了。✨

## 2. 安装

全局安装：

```shell
npm install -g batch-run-local
```

安装完成之后，就可以使用 `br` 命令了：

```shell
br
```

直接输入 `br`，会出现一个交互菜单，大概就是让你选择：

- 打开可视化操作页面
- 启动项目
- 查看项目
- 新增项目
- 删除项目
- 设置 Git Bash 路径

这一步还是很友好的，不用一上来就背命令。像我这种记忆力随缘型选手，很需要这种菜单。🥲

## 3. 添加项目

先运行：

```shell
br add
```

它会让你输入两个东西：

```text
项目名称
项目根路径
```

比如我想把博客项目加进去，就可以这样理解：

```text
项目名称：my_blog
项目根路径：C:\wang\project\my_blog
```

这里的项目根路径就是有 `package.json` 的那个目录。因为后面工具会去这个目录下面实时读取 `package.json` 里的 `scripts`。

也就是说，你不用在 `batch-run-local` 里再手动维护一份 `dev`、`build`、`preview` 之类的命令。项目里的 `package.json` 怎么写，它就怎么展示。这个设计我觉得挺舒服的，少一份配置，就少一个出错的地方。😎

添加成功后，它会提示：

```text
已添加项目: xxx
```

这就算把项目收进“启动清单”里了。✅

## 4. 查看项目

如果项目加多了，想看看当前都有哪些，可以执行：

```shell
br list
```

它会列出项目名称、项目 id、项目路径，还会告诉你配置文件放在哪里。

这里有一个我很喜欢的小点：它的配置是存在用户电脑的应用配置目录里，不会写进被管理项目。也就是说，它不会突然跑到你的业务项目里加一些奇奇怪怪的配置文件。干干净净，很安心。🧼

## 5. 启动项目

重点来了。

执行：

```shell
br run
```

接下来它会做两件事：

1. 先让你选择一个已经添加过的项目
2. 再读取这个项目的 `package.json scripts`，让你选择要执行哪个脚本

比如项目里有：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
```

那你在命令行里就能直接看到 `dev`、`build`、`preview` 这些选项。

选择 `dev` 之后，它会打开一个新的 Git Bash 窗口，并在项目目录下执行：

```shell
npm run dev
```

这个体验就比较像：

“我要启动这个项目。”

“好的，Git Bash 已经给你打开了，命令也跑起来了。” 🤖

不用自己找路径，不用自己复制命令，不用担心终端当前目录不对。对于本地多项目开发来说，真的能少很多机械操作。

## 6. Web 管理页面

如果你不想一直在命令行里选来选去，也可以打开它自带的本地 Web 页面：

```shell
br ui
```

执行后，它会在本地启动一个服务，并自动打开浏览器。

<img src='./imgs/ui01.png'/>

页面里可以做这些事：

- 查看已经配置的项目 📋
- 新增项目 ➕
- 删除项目 🗑️
- 读取项目的 `package.json scripts` 📦
- 点击按钮执行对应脚本 🖱️
- 打开项目目录 📁

这个 Web 页面就更适合“我今天要同时看好几个项目”的场景。

比如左边选中 `my_blog`，右边出现 `dev`、`build`、`preview`。想跑哪个就点哪个，点完它会打开 Git Bash 执行命令。

有点像一个本地版的项目启动面板。虽然它没有花里胡哨的大屏动画，但够直接，够实用。👍

## 7. Git Bash 路径配置

这个工具当前主要面向 Windows + Git Bash。

它会优先自动检测常见路径：

```text
C:\Program Files\Git\git-bash.exe
C:\Program Files (x86)\Git\git-bash.exe
```

如果你的 Git Bash 装在别的位置，或者自动检测失败，可以运行：

```shell
br config
```

然后手动填写 `git-bash.exe` 的路径。

这里也可以顺便配置 Web 服务端口，默认端口是：

```text
1234
```

如果端口被占用了，改一下就行。🔧

## 8. 删除项目

项目不用了，也可以删掉：

```shell
br remove
```

它会让你从列表里选择要删除的项目。

注意，这里删除的是 `batch-run-local` 里的项目配置，不会删除你的真实项目文件夹。也就是只是从“小本本”里划掉，不会把你电脑里的代码干掉。这个很重要，放心程度 +1。😌

## 9. 常用命令速查

```shell
br          # 打开主菜单
br add      # 新增项目
br list     # 查看项目
br run      # 选择项目和 scripts 后执行
br ui       # 打开本地 Web 管理页面
br config   # 设置 Git Bash 路径和 Web 端口
br remove   # 删除项目
```

如果日常只记三个，我觉得是这三个：

```shell
br add
br run
br ui
```

一个负责收项目，一个负责跑项目，一个负责打开可视化页面。够用了。🎯

## 10. 使用体验

它解决的不是那种“惊天动地”的大问题，而是开发日常里很碎、很烦、很重复的小动作。

以前启动一个项目，我的流程可能是：

打开文件夹 👉 找项目 👉 复制路径 👉 打开 Git Bash 👉 `cd` 过去 👉 翻 `package.json` 👉 找脚本 👉 输入命令 👉 回车

现在变成：

```shell
br run
```

然后选一下。

或者更懒一点：

```shell
br ui
```

浏览器里点一下。

这种感觉就像把桌面上乱七八糟的数据线都理顺了。不是不能用，但理顺之后心情会好很多。🌈

## 11. 最后

`batch-run-local` 更像是一个给本地开发用的小工具：轻量、直接、围绕 `package.json scripts` 做事情。

如果你本地只有一个项目，可能感受不明显；但如果你经常在多个前端项目、Node 服务、之间来回切，它就会变成一个很顺手的小帮手。

全局安装，一次添加，之后启动项目就不用再到处找路径了。

这大概就是工具存在的意义吧：把重复的小麻烦一点点拿掉，把时间留给真正要写的代码。✨
